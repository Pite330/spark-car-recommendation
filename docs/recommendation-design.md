# 推荐算法设计

## 1. 设计目标

算法优先保证可解释、稳定和容易验证。MVP 不训练复杂模型，而是使用硬性筛选与加权评分生成推荐结果。

## 2. 输入条件

- 必填：预算下限、预算上限；
- 可选：车身类型、能源类型、品牌、使用场景、座位数、最低月销量；
- 未填写条件不参与筛选或评分。

## 3. 硬性筛选

默认将预算、车身类型和能源类型视为硬性条件：

1. 车型价格区间与用户预算有交集；
2. 用户选择车身类型时，车型必须匹配；
3. 用户选择能源类型时，车型必须匹配；
4. 用户明确最低座位数时，车型不得低于要求。
5. 用户明确最低月销量时，销量缺失或低于阈值的车型不进入候选。

## 4. 结果不足时的放宽策略

当候选少于 3 款时，按以下顺序放宽：

1. 忽略品牌偏好；
2. 忽略使用场景标签；
3. 将预算上限扩大最多 10%；
4. 仍无结果时返回空列表，不擅自更换能源类型或车身类型。

返回结果中必须包含 `relaxed_conditions`，明确告诉用户放宽了哪些条件。

当前实现先在核心候选中尝试品牌与场景约束；只有约束后仍有至少 3 款才保留，否则记录对应放宽。预算扩张只改变上限，不改变预算下限、车身、能源或最低座位数。

## 5. 综合评分

初始权重如下：

| 因素 | 权重 | 说明 |
| --- | ---: | --- |
| 预算匹配度 | 35% | 价格越接近预算中间值，得分越高 |
| 车身类型 | 20% | 用户选择后完全匹配得满分 |
| 能源类型 | 20% | 用户选择后完全匹配得满分 |
| 使用场景 | 15% | 根据已有参数和场景标签匹配 |
| 品牌偏好 | 5% | 品牌匹配时加分 |
| 销量或热度 | 5% | 只作为辅助排序因素 |

用户未填写某项或数据缺失时，删除该评分项，并把剩余权重重新归一化为 100%。

页面车型对比不再把缺失较多、区分度有限的字段强行归一化成固定雷达维度。主图只比较参考起售价和同期月销量两个可直接解释的绝对值；销量缺失时显示“暂无”且不绘制销量柱，不使用 0 代替未知值。座位数、续航、油耗、功率、车型年款、在售配置数等字段保留在参数表中，并且只在至少一款车型存在真实值时显示对应行。

## 6. 预算匹配示例

设用户预算中点为 `B`，车型价格中点为 `P`，预算跨度为 `R`：

```text
price_score = max(0, 1 - abs(P - B) / max(R, 1))
```

最终总分统一换算为 0—100。若多个车型分数相同，依次按数据完整度、热度和车型名称稳定排序。

## 7. 推荐原因生成

算法输出贡献最高的 2—3 个匹配因素，例如：

- 价格位于预算中段；
- 符合纯电 SUV 偏好；
- 续航满足设定的城市通勤需求。

模板模块将这些依据组合为自然语言。大模型只能改写这些依据，不得添加新的车辆事实。

## 8. 可测试性

至少准备以下测试：

- 相同输入重复运行，排序一致；
- 预算上下限颠倒时拒绝请求；
- 某个可选字段全为空时仍能推荐；
- 无候选时不返回无关车型；
- 放宽预算时明确记录；
- 大模型失败时车型、得分和模板原因保持完整。
