# 基于Spark的汽车大数据分析与决策支持平台——MVP方案

## 一、MVP目标

在有限实训周期内完成一个可运行、可演示的消费者购车推荐闭环：消费者输入预算及购车偏好，系统从汽车数据集中筛选并计算推荐结果，返回 3—5 款车型及可解释的推荐原因，并支持查看关键参数和简单车型对比。

MVP 的重点不是建设完整汽车平台，而是证明以下链路能够跑通：

> 汽车原始数据 → Spark 数据处理 → 用户输入条件 → 车型筛选与评分 → 推荐车型与原因 → 页面展示

## 二、目标用户与使用流程

目标用户为近期有购车需求、但缺少汽车专业知识的普通消费者。

基本使用流程如下：

1. 用户进入购车推荐页面；
2. 输入预算上限和下限；
3. 选择车型、能源类型、品牌或使用场景等偏好；
4. 点击“开始推荐”；
5. 系统返回 3—5 款推荐车型；
6. 用户查看每款车的关键参数、匹配得分和推荐原因；
7. 用户选择其中 2—3 款车型进行简单对比。

## 三、MVP功能范围

### 必须完成

1. 一份结构统一、可以运行 Spark 任务的汽车数据集；
2. 使用 PySpark 完成数据清洗、字段转换和特征处理；
3. 一个包含预算、车型和能源类型等条件的输入页面；
4. 基于硬性筛选与加权评分的推荐算法；
5. 返回 3—5 款车型及结构化推荐原因；
6. 推荐结果卡片和 2—3 款车型的关键参数对比；
7. 至少一项能够体现 Spark 处理过程的运行结果或统计图表；
8. 大模型不可用时仍能正常生成推荐结果和模板化推荐原因。

### 可以完成

- 接入大模型，将结构化推荐依据改写为更自然的说明；
- 增加价格区间、能源类型或车型分布图；
- 支持用户调整条件后重新推荐；
- 增加推荐结果导出或截图功能。

### 明确不做

- 注册、登录、用户中心及权限管理；
- 管理员后台和通用数据增删改查系统；
- 面向厂商的销量预测、生产规划或营销决策；
- 在线支付、订单、试驾预约和经销商对接；
- 复杂深度学习模型训练；
- 让大模型直接代替推荐算法。

## 四、最小数据字段

汽车数据集至少包含以下字段：

| 字段 | 用途 |
| --- | --- |
| 车型名称 | 展示推荐结果 |
| 品牌 | 品牌偏好匹配 |
| 指导价格 | 预算筛选与价格匹配 |
| 车身类型 | 轿车、SUV、MPV 等偏好匹配 |
| 能源类型 | 燃油、纯电、插混等偏好匹配 |
| 座位数 | 家庭或多人出行场景匹配 |
| 续航里程或油耗 | 通勤、长途等场景匹配 |
| 销量或热度 | 作为推荐排序的辅助指标 |

数据缺少某些可选字段时，推荐算法应自动取消对应评分项，不能用虚构数据补齐。

## 五、推荐算法方案

推荐分为“硬性筛选”和“综合评分”两步。

### 1. 硬性筛选

- 车辆价格应在用户预算范围内或接近预算上限；
- 车型符合用户选择的车身类型；
- 能源类型符合用户选择；
- 用户未选择的条件不作为限制。

### 2. 综合评分

候选车辆可以按照以下示例权重进行评分：

| 评分因素 | 建议权重 |
| --- | ---: |
| 预算匹配度 | 35% |
| 车型匹配度 | 20% |
| 能源类型匹配度 | 20% |
| 使用场景适配度 | 15% |
| 品牌偏好 | 5% |
| 销量或热度 | 5% |

权重可以根据实际数据字段调整，但必须在报告中说明评分依据。推荐原因直接读取得分较高的匹配项，例如“价格位于预算中段”“符合纯电 SUV 偏好”“续航适合日常通勤”，以保证推荐结果可解释。

## 六、大模型在MVP中的位置

大模型属于可选增强功能，不影响 MVP 主链路验收。

程序先产生如下结构化信息：

```json
{
  "model": "示例车型A",
  "matched_factors": ["价格符合预算", "纯电SUV", "续航适合城市通勤"],
  "score": 91
}
```

接入大模型时，只允许其根据这些已知信息生成一段自然语言说明，不允许补充数据集中没有的配置或自行更换推荐车型。接口超时、额度不足或网络不可用时，系统立即退回模板化说明，保证现场演示稳定。

## 七、最小技术方案

- 数据处理：Python、PySpark；
- 数据存储：CSV 或 Parquet，课程要求需要时同步保存至 HDFS；
- 推荐算法：Python 规则筛选与加权评分；
- 页面与轻量接口：Flask、HTML、CSS、JavaScript；
- 数据可视化：ECharts；
- 大模型：兼容 OpenAI 接口格式的模型 API，可选；
- 版本管理：Git。

Flask 只负责加载处理后的数据、接收推荐条件和返回结果，不建设复杂后台业务。

## 八、四人分工与职位

| 成员职位 | 主要职责 | 主要交付物 |
| --- | --- | --- |
| 项目负责人兼产品与数据协调 | 明确功能范围和数据字段；组织任务拆分与进度同步；整理汽车原始数据；负责模块集成、项目文档和答辩流程 | 需求说明、数据字典、进度计划、集成版本、项目报告和答辩材料 |
| Spark数据工程师 | 编写 PySpark 数据清洗程序；完成去重、缺失值处理、字段转换和特征标准化；输出推荐使用的标准数据集及统计结果 | Spark处理脚本、标准数据集、处理结果说明、运行截图 |
| 推荐算法与大模型工程师 | 设计硬性筛选和加权评分算法；生成结构化推荐依据；实现推荐理由模板；在主流程完成后接入可选大模型并实现失败回退 | 推荐算法模块、评分权重说明、推荐原因模块、大模型可选接口 |
| 前端可视化与测试工程师 | 实现条件输入页面、推荐结果卡片和车型对比页面；使用 ECharts 展示必要图表；编写测试用例并记录问题 | 前端页面、可视化图表、车型对比功能、测试记录和演示页面 |

四人共同参与最终联调、演示数据准备和答辩。项目负责人负责确认接口格式，其他成员分别对数据、算法和页面模块负责，避免多人同时修改同一核心文件。

## 九、MVP验收标准

1. 用户能够输入预算，并至少选择车型或能源类型中的一项；
2. 系统能够稳定返回 3—5 款符合条件的车型；
3. 每款车型至少展示价格、车型、能源类型和两条推荐依据；
4. 推荐结果由可复现的筛选与评分算法产生，相同输入能够得到一致结果；
5. 用户能够对比 2—3 款推荐车型的关键参数；
6. 能够展示 Spark 数据清洗或统计分析的实际运行过程和结果；
7. 关闭大模型接口后，推荐功能仍然可以完整运行；
8. 系统不包含登录、后台管理和厂商决策等范围外功能。

## 十、最终演示脚本

1. 展示原始汽车数据和 Spark 清洗结果；
2. 输入一个明确购车需求，例如“预算 15—20 万、纯电 SUV、主要用于城市通勤”；
3. 展示系统返回的 3—5 款车型、匹配得分和推荐原因；
4. 选择两款车型进行参数对比；
5. 演示大模型生成的自然语言推荐说明；
6. 关闭或跳过大模型接口，再次展示模板化推荐原因，证明核心功能不依赖外部模型。
