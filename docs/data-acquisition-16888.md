# 16888 数据采集与使用说明

## 1. 数据源结论

项目当前数据源为 [16888 车主之家](https://www.16888.com/)，默认原始输入是 `data/raw/16888_full/tables/`，抓取日期为 `2026-07-11`。

完整抓取包含 1620 个车系、19482 个配置款、3643134 条配置参数和 29859 条月销量。Spark 从中形成 1746 条“车系 × 能源类型”聚合记录，删除 2 条无有效厂商名称的记录后输出 1573 个有效车系、1744 条推荐记录；一个车系有多种动力形式时会拆成多条记录。

## 2. 采集入口与范围

脚本 `scripts/fetch_16888_full_options.py` 使用以下公开入口：

| 数据 | 入口 | 用途 |
| --- | --- | --- |
| 车身级别车系列表 | `https://auto.16888.com/form.html` | 获取公开可见的各级别车系列表 |
| 新能源车系列表 | `https://auto.16888.com/ev.html` | 补充纯电、插混、混动和增程车系 |
| 公开参数接口 | `https://www.16888.com/auto/auto.php?mod=auto&extra=scapi&sid={sid}&decade=0` | 获取车系配置、价格、厂商、级别、车身、能源、座位、年款和功率 |
| 车系参数页 | `https://www.16888.com/{sid}/options/` | 作为参数接口来源页和记录证据 |
| 车系销量页 | `https://xl.16888.com/s/{sid}/` | 获取最近月度车系销量及统计周期 |

采集范围是抓取时公开可见的 1620 个车系，不是持续同步的全站镜像。脚本串行请求、默认请求间隔为 1 秒，不绕过登录、验证码或访问限制；日常演示只读取本地缓存。

## 3. 原始字段

| 字段 | 说明 |
| --- | --- |
| `source_key` | `series_id-energy_type` 组成的稳定业务键 |
| `series_id` | 16888 车系 ID |
| `model_name`、`brand` | 车系名称和品牌/厂商归一名称 |
| `price_min_wan`、`price_max_wan` | 页面展示的万元指导价区间 |
| `level_name`、`body_structure` | 原始级别和车身结构 |
| `energy_type` | 归一为燃油、纯电、插混、增程、混动 |
| `seats`、`horsepower`、`model_year` | 参数配置聚合后的代表值 |
| `range_km`、`fuel_consumption` | 来源存在时保留，不存在时为空 |
| `sales`、`sales_period` | 最近月度车系销量和月份 |
| `trim_count` | 当前车系能源分组内参与聚合的配置数 |
| `source_url`、`options_url`、`sales_url` | 车系、参数和销量来源页 |
| `captured_at` | 抓取日期 |

## 4. 聚合与 Spark 处理

Spark 阶段按“车系 + 能源类型”聚合配置。价格取该分组所有有效配置的最小值和最大值；厂商和级别读取完整参数表；座位取可用最大值，功率和油耗取可用中位数，年款取最新值。月销量是车系级数据，所以同一车系的不同能源记录共享最近月销量。

PySpark 清洗任务负责：

1. 校验必需字段并删除无效记录；
2. 将原始级别/车身映射为轿车、两厢车、SUV、MPV 等标准枚举；
3. 校验座位、油耗、功率、年款和销量边界；
4. 生成稳定 `car_id`、价格中点、场景标签和数据完整度；
5. 对月销量执行 `log(sales + 1)` 归一化，生成 0—1 热度；
6. 输出 CSV、Parquet、统计结果和带输入 SHA-256 的运行元数据。

运行命令：

```bash
make data-full  # 主动联网更新完整原始数据
make spark      # 从完整原始表重建标准数据
```

日常演示不需要运行 `make data-full`，直接使用本地完整快照和已生成的 Spark 结果即可。

## 6. 更新验收

每次重新采集后至少执行：

```bash
make spark
make test
```

并检查：

- `data/processed/metadata.json` 中 `data_source` 为 `16888.com 车主之家`；
- 输入/输出行数合理，必需字段缺失为 0；
- 演示条件 `15—20 万 / SUV / 纯电 / 城市通勤 / 5 座` 能返回 3—5 款；
- 相同输入重复推荐顺序一致；
- `git diff -- data/raw data/processed` 中不存在明显页面解析错位。

## 7. 使用边界

- 价格是抓取时公开展示的指导价，不是实时成交价；
- 销量是站点公开的车系月销量，不等同于官方上牌量；
- 数据覆盖抓取时公开可见车系，不代表实时市场全量车型；
- 页面结构变化可能导致字段失效，采集脚本需要随之更新；
- 公开仓库、扩大采集或商业使用前，应重新核查来源站点条款、版权和数据授权；
- `.env`、API Key、Cookie、验证码和个人凭据不得写入数据文件或提交 Git。

## 8. 完整配置参数与销量历史

消费者推荐快照只保留少量聚合字段。厂商侧探索性分析使用独立脚本：

```bash
# 先试采前 20 个车系
make data-full-pilot

# 确认字段和磁盘空间后，再采集当前分类页发现的全部在售车系
make data-full

# 只采集指定车系；57232 是奥迪 A6L 车系 ID
.venv/bin/python scripts/fetch_16888_full_options.py --series-id 57232

# 中途停止采集后，只根据现有缓存重建分析表，不再联网
.venv/bin/python scripts/fetch_16888_full_options.py --build-only
```

默认输出目录是 `data/raw/16888_full/`，该目录由 `.gitignore` 排除，不提交大体积原始响应。脚本默认设置 1 秒请求间隔；每个车系的参数 JSON 和销量 HTML 会先写入 `responses/`，中断重跑时复用已有缓存。使用 `--force-refresh` 才会重新请求已经缓存的车系。

规范化结果位于 `tables/`：

- `series.csv`：车系及分类提示；
- `trims.csv`：每个在售配置款的标识和基础元数据；
- `parameter_definitions.csv`：参数分组、名称和单位字典；
- `trim_parameters.csv.gz`：完整“配置款 × 参数”压缩长表；
- `series_month_sales.csv`：销量页面公开的车系月度历史；
- `manifest.json`：行数、采集时间、请求间隔和失败清单。

`decade=0` 对应参数页的“全部在售”，因此这份数据不是历史配置档案。月销量是车系级而不是配置款级；分析参数与销量关系前，需要先把同车系配置款聚合为数值范围、标配率或选配率，并将结论表述为关联而不是因果影响。
